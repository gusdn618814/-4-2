# 과제 2) 피보나치 및 GCD 시간 복잡도 프로파일링 보고서

## 1. 과제 개요
- **목표**: 재귀적 피보나치 수열 $F(n)$을 구현하고, 인접한 두 피보나치 수 $F(n)$과 $F(n-1)$의 GCD를 계산하면서 CPU 시간과 메모리 사용량을 측정한다.  
- **측정 범위**: $n = 5$부터 $5$씩 증가시켜 $n = 45$까지.  
- **측정 항목**: 실행 시간(CPU Time), 메모리 점유량, GCD 결과값.

## 2. 알고리즘 및 복잡도 분석 (Big‑O)

### 2.1. 재귀적 피보나치
- **구현**: `F(n) = F(n‑1) + F(n‑2)` (Base case: `F(0)=0, F(1)=1`)  
- **시간 복잡도**: 호출이 이진 트리 형태로 전개되므로 **$O(2^n)$** (정확히는 $\phi^n$, $\phi≈1.618$).  
- **공간 복잡도**: 재귀 깊이가 $n$에 비례 → **$O(n)$**.

### 2.2. 유클리드 GCD
- **특성**: 인접 피보나치 수 $(F_n, F_{n-1})$는 서로소이므로 최악의 경우를 제공한다.  
- **시간 복잡도**: 입력 크기(비트 수)에 대해 **$O(\log n)$**.

## 6. 분석 및 결론
1. **CPU 사용량**  
   - $n$이 30을 넘어가면서 실행 시간이 급격히 증가한다. 이는 재귀 피보나치가 $O(2^n)$ 의 지수적 복잡도를 갖는다는 이론과 일치한다.  

2. **메모리 사용량**  
   - 재귀 깊이가 $n$에 비례해 스택이 늘어나지만, 전체 메모리 점유량은 `O(n)` 수준으로 CPU 시간에 비해 완만하게 증가한다.  

3. **GCD**  
   - 인접 피보나치 쌍은 최악의 경우에도 Euclidean 알고리즘이 **극히 빠르게** (`O(log n)`) 결과를 반환한다는 것을 확인했다.  

> **결론**: 재귀적으로 구현된 피보나치 함수는 실용적인 입력 범위(예: $n>30$)에서는 비효율적이며, 동적 프로그래밍(DP)이나 메모이제이션을 사용해야 한다는 점을 실험적으로 증명하였다. 반면 GCD는 입력 크기에 거의 영향을 받지 않으며, 알고리즘 자체는 최적임을 확인하였다.
